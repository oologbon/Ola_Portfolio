Project 2: Log Monitoring with Bash Script

April 10, 2023

Workflow:

Configure Bash scripts on the Linux machine to monitor the Apache web server access logs, especially login attempts. Using the Windows machines on the network to access the Apache webserver to test if the bash scripts would capture the events. The bash script will also monitor HTTP status code in the web server. Monitoring HTTP status codes in web server logs could be very helpful in understanding the behavior of the web server and diagnosing any problems. HTTP status codes like 200 and 404 can give you insight into the success or failure of requests to your server.

Programming:

For my monitoring and alert system, I wrote a script that reads and filters an Apache access log file, and for each IP in a predefined list, it counts the number of connections and the number of HTTP 404 status codes. Monitoring the HTTP 404 status code is to detect unsuccessful attempt to the web server. The script writes these numbers to a log file, and if the number of 404 errors for any IP exceeds a predefined threshold, it writes an alert to the log file.

In the above script, I first define a set of variables: filename: This is the location of the apache access log file. target_ips: This is an array of IP addresses that the script will check in the log file. log_file: This is the file where the results of the script will be written. threshold: This is the limit of 404 errors that can occur before an alert is triggered.

Then I use a “for” loop to go through each IP in the “target_ips” array. Inside the loop, I first output a line to the “log_file” indicating that the script is processing the current IP. I then use a grep command to count the number of lines in the log file that contain the current IP. This count is stored in the count variable. To loop through the ips in the access log and print the number of occurrences, I used a conditional statement:

If count is 0 (indicating that there are no lines in the log file with the current IP), it outputs a message to the “log_file” indicating that no matches were found for the current IP. However, If count is not 0, it continues to: Get the current date and time and store it in the “timestamp” variable. Output a line to the “log_file” indicating the number of connections for the current IP. Use grep to count the number of lines in the log file that contain both the current IP and “404” errors. This count is stored in the “error_count” variable. Output a line to the “log_file” indicating the number of 404 errors for the current IP. Then to print an alert if the count is greater then the number set in the “threshold” variable, I used another if statement: If error_count is greater than the threshold, it outputs an alert to the log_file indicating that the number of 404 errors exceeded the threshold. The script continues to loop through all the IPs in the “target_ips” array.

Documentation:

For the sake of documentation, I appended all the outputs in a cron job to a log file by using the » operator. This operator will append the output of the command to the specified file. The cron job is scheduled to run my bash script and append all the alerts to a file for easy documentation.

Expected Output

In the context of my log monitoring process, the expected output includes the following:

The number of detected failed login attempts, including the time they occur. The IP addresses linked with these unsuccessful efforts.

These data points are instrumental, not just in identifying potential threat actors or compromised accounts, but also in understanding the progression and nature of an attack. For instance, a high volume of failed attempts on the web server could indicate an ongoing reconnaissance attack. For instance, it could indicate a web directory enumeration reconnaissance attack. This kind of pattern suggests the attacker might be probing for vulnerabilities to exploit. So, this information provides valuable insight into the attacker’s tactics and possible intentions, which is critical in forming an effective defence strategy.

Unusual Behavior

A high number of failed login attempts, known malicious IPs attempting to log in, and unusual login patterns constitute flags for alerts. For instance, in the following result, one could see a high number of attempts from ips 172.16.14.50 and 172.16.14.54. Both ips attempted to access the webserver more than ten times, way above the set threshold. This kind of behavior constitutes an unusual behaviour worthy of attention.
